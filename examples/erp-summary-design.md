## 1. 引言

### 1.1 编写目的

本文档面向系统架构师、模块设计人员、开发与测试团队、运维及项目管理人员，旨在对 ERP 系统进行总体性、结构性的设计描述。其目的在于：

1. 确定系统的总体架构、技术选型与模块划分，作为详细设计阶段的输入与约束。
2. 明确各业务领域（采购、销售、库存、财务、生产、主数据）的职责边界与协作关系，避免功能重叠与遗漏。
3. 定义核心数据模型、内部模块接口与外部系统接口契约，为后续接口开发、数据建库提供依据。
4. 规定非功能性指标（性能、安全、可用性、可扩展性）及 ERP 特有的多组织、多币种、审计追溯设计原则。

本文档不展开到字段级表结构、类设计、算法伪码等详细设计内容，这些由各模块《详细设计说明书》承接。

### 1.2 项目背景与范围

#### 1.2.1 背景

企业现有业务系统呈"烟囱式"分布：采购、销售、仓储、财务、生产分别由独立软件或 Excel 管理，数据不互通，导致库存不准、账实不符、成本核算滞后、跨部门协同效率低。本项目建设统一的一体化 ERP 平台，实现"业务一体化、财务业务一体化、产供销一体化"。

#### 1.2.2 范围

| 范围 | 内容 |
|---|---|
| 纳入范围 | 主数据管理、采购管理（P2P）、销售管理（O2C）、库存管理、生产管理（MRP/BOM）、财务管理（总账/应收/应付/成本）、系统管理与权限、对外集成（银行/税务/电商/MES） |
| 暂不纳入 | 人力资源（HR）、固定资产全生命周期、商业智能（BI）独立平台、跨境多税制合规（一期仅支持中国大陆税制，预留扩展） |
| 用户对象 | 采购员、销售员、仓管员、计划员、生产主管、财务（出纳/会计/成本/主管）、系统管理员、企业管理层 |

## 2. 总体设计

### 2.1 设计目标与原则

| 原则 | 说明 |
|---|---|
| 业财一体 | 业务单据驱动财务凭证自动生成，账实同源、单据可追溯 |
| 模块解耦 | 领域边界清晰，模块间通过标准接口/领域事件协作，可独立演进 |
| 主数据唯一 | 物料、客户、供应商、组织统一建模、统一编码、全局共享 |
| 多组织支持 | 支持集团-法人-业务单元多层组织，数据按组织隔离与共享 |
| 可配置可扩展 | 业务规则、单据流、审批流可配置；预留二开扩展点 |
| 安全可审计 | 全程权限控制与操作留痕，关键数据变更可追溯 |

### 2.2 总体架构（三视图）

#### 2.2.1 分层架构视图

系统采用经典分层 + 领域驱动设计（DDD）思想，自上而下分为五层：

```
┌───────────────────────────────────────────────────────────┐
│  接入层 Access      Web端 / 移动端 / 开放API / 第三方对接      │
├───────────────────────────────────────────────────────────┤
│  网关层 Gateway     统一认证鉴权 · 路由 · 限流熔断 · 灰度      │
├───────────────────────────────────────────────────────────┤
│  应用层 Application 业务编排 · 流程引擎 · DTO组装 · 事务协调   │
├───────────────────────────────────────────────────────────┤
│  领域层 Domain      采购 销售 库存 生产 财务 主数据 (领域模型) │
│                     领域服务 · 领域事件 · 聚合根 · 业务规则    │
├───────────────────────────────────────────────────────────┤
│  基础设施层 Infra   持久化 · 缓存 · 消息 · 文件 · 集成适配器   │
└───────────────────────────────────────────────────────────┘
        横切关注点：日志 · 审计 · 配置中心 · 权限 · 多租户上下文
```

- **接入层**：多端统一，Web 端供后台操作，移动端供审批/扫码出入库，开放 API 供电商、MES、第三方对接。
- **网关层**：统一入口，承担认证（OAuth2/JWT）、鉴权、路由、限流、熔断、灰度发布。
- **应用层**：用例编排，组织跨领域调用，管理分布式事务边界（Saga / TCC），不含核心业务规则。
- **领域层**：业务核心，各领域以聚合根为中心封装业务规则，通过领域事件向外发布状态变化。
- **基础设施层**：技术细节实现，包含 ORM 持久化、缓存、消息中间件、外部系统适配器。

#### 2.2.2 技术架构视图（技术选型）

| 层次 | 选型 | 说明 |
|---|---|---|
| 前端框架 | Vue 3 + TypeScript + Element Plus；移动端 uni-app | 组件化、低代码表单引擎支撑单据快速搭建 |
| 后端框架 | Java 17 + Spring Boot 3.x + Spring Cloud Alibaba | 微服务体系成熟，生态完善 |
| 服务注册/配置 | Nacos | 服务发现 + 动态配置中心 |
| API 网关 | Spring Cloud Gateway + Sentinel | 路由、限流、熔断降级 |
| 认证授权 | Spring Authorization Server（OAuth2.1）+ JWT | 统一身份认证（SSO） |
| 关系数据库 | PostgreSQL 15（主）/ MySQL 8 备选；分库分表用 ShardingSphere | 事务型核心数据，强一致 |
| 缓存 | Redis 7（集群） | 会话、热点主数据、分布式锁、序列号 |
| 消息中间件 | RocketMQ / Kafka | 领域事件、异步解耦、削峰、最终一致 |
| 搜索引擎 | Elasticsearch 8 | 单据/主数据全文检索、报表聚合 |
| 工作流引擎 | Flowable / Camunda | 审批流、单据流转 |
| 报表 | 自研报表中心 + JimuReport；OLAP 用 Doris/ClickHouse | 经营分析、明细账查询 |
| 分布式事务 | Seata（AT/TCC/Saga） | 跨服务一致性 |
| 文件存储 | MinIO / 对象存储 | 附件、电子凭证、电子发票 |
| 容器与编排 | Docker + Kubernetes | 弹性伸缩、滚动发布 |
| 可观测性 | Prometheus + Grafana + SkyWalking + ELK | 指标、链路追踪、日志 |
| 定时调度 | XXL-Job | MRP 运算、对账、利息计提等批处理 |

```

                      │ HTTPS
                ┌─────▼─────┐
                │  负载均衡 LB │ (Nginx/SLB, 双活)
                └─────┬─────┘
                ┌─────▼─────┐
                │  API 网关集群 │
                └─────┬─────┘
        ┌─────────────┼─────────────┐
   ┌────▼────┐  ┌────▼────┐   ┌────▼────┐
   │ 微服务集群 │  │ 微服务集群 │ … │ 集成服务  │  (K8s Pod, 多副本)
   │ 采购/销售  │  │ 库存/生产  │   │ 银行/税务 │
   │ 财务/主数据 │  │           │   │ 电商/MES │
   └────┬────┘  └────┬────┘   └────┬────┘
        └─────────────┼─────────────┘
   ┌──────┬───────────┼──────────┬────────┐
┌──▼──┐┌──▼──┐    ┌──▼──┐   ┌──▼──┐  ┌──▼──┐
│ PG  ││Redis│    │RocketMQ│ │ ES  │  │MinIO│
│主从+ ││集群 │    │集群    │ │集群 │  │集群 │
│读写分离│       └─────┘   └─────┘  └─────┘
└─────┘
```

- 应用层无状态，K8s 多副本横向扩展；数据库主从 + 读写分离，核心库异地灾备。
- 按"两地三中心"规划：生产中心 + 同城灾备 + 异地灾备，RPO ≤ 5 分钟，RTO ≤ 30 分钟。

### 2.3 模块总览与边界

| 模块 | 职责 | 边界（不负责） |
|---|---|---|
| 主数据 | 物料、客户、供应商、组织、计量单位、币种统一管理 | 不处理业务交易 |
| 采购 P2P | 请购、询比价、采购订单、收货、采购退货、发票校验 | 入库实物增减由库存负责，付款由 AP 负责 |
| 销售 O2C | 报价、销售订单、发货通知、销售退货、开票申请 | 实物出库由库存负责，收款由 AR 负责 |
| 库存 | 出入库、调拨、盘点、库存账、批次/序列号、库龄 | 不决定采购/销售业务，仅响应单据执行实物增减 |
| 生产 | BOM、工艺路线、MRP 运算、生产订单、领料/完工入库 | 车间设备级执行交由 MES |
| 财务 | 总账 GL、应收 AR、应付 AP、成本核算 | 不发起业务，凭证由业务单据驱动生成 |
| 系统管理 | 用户、角色、权限、组织、审批流、参数配置 | — |

## 3. 功能模块设计

### 3.1 基础主数据管理

**职责**：作为全系统数据基石，统一定义并维护被各业务模块共享的静态数据。

| 子功能 | 说明 |
|---|---|
| 物料主数据 | 物料编码、规格、计量单位、物料组、采购/销售/库存/生产属性、批次/序列号管控标识 |
| 客户主数据 | 客户编码、信用额度、结算条件、收货地址、税务信息、所属组织 |
| 供应商主数据 | 供应商编码、采购组、付款条件、银行账户、税号、准入状态 |
| 组织主数据 | 集团/法人/利润中心/采购组织/销售组织/库存地点/工厂多层级 |
| 通用主数据 | 计量单位及换算、币种与汇率、税码、会计科目、银行 |

**边界**：主数据仅维护"是什么"，不参与"做什么"（交易）。主数据变更通过领域事件广播，业务模块订阅刷新缓存。

### 3.2 采购管理（P2P）

| 子功能 | 说明 |
|---|---|
| 请购管理 | 来自 MRP 建议、库存补货点、手工请购，进入审批 |
| 询价比价 | 多供应商报价比对，确定供应商与价格 |
| 采购订单 | 生成 PO，含数量、价格、税率、交期、付款条件，审批后下达 |
| 收货管理 | 按 PO 收货，生成收货单触发库存入库与暂估 |
| 采购退货 | 质量不合格退货，冲减库存与暂估/应付 |
| 发票校验 | 三单匹配（PO+收货+发票），匹配通过生成应付 |

**与其他模块协作**：收货 → 触发库存入库；发票校验通过 → 生成 AP 应付单 + GL 凭证。

### 3.3 销售管理（O2C）

| 子功能 | 说明 |
|---|---|
| 报价管理 | 客户报价、有效期、价格策略 |
| 销售订单 | 客户、商品、数量、价格、税、交期；下单时校验信用与可用库存（ATP） |
| 发货管理 | 生成发货通知 → 拣货 → 出库，触发库存减少 |
| 销售退货 | 退货入库，冲减收入与应收 |
| 开票申请 | 按发货或订单开票，生成销项发票数据 |

**与其他模块协作**：销售订单占用库存（ATP）；发货 → 触发库存出库 + 结转销售成本；开票 → 生成 AR 应收 + GL 凭证。

### 3.4 库存管理

| 子功能 | 说明 |
|---|---|
| 入库管理 | 采购入库、生产完工入库、退货入库、其他入库 |
| 出库管理 | 销售出库、生产领料出库、调拨出库、其他出库 |
| 库存调拨 | 跨库存地点、跨组织调拨，含在途库存 |
| 盘点管理 | 周期盘点/动态盘点，盘盈盘亏生成调整单与凭证 |
| 批次与序列号 | 批次属性、保质期、序列号全程追溯 |
| 库存账与库龄 | 实时结存、可用量、库龄分析、呆滞预警 |

**边界**：库存只对"实物数量与状态"负责，被业务单据驱动，不主动产生业务决策。计价（移动加权/先进先出/标准成本）与成本模块联动。

### 3.5 生产管理（MRP / BOM）

| 子功能 | 说明 |
|---|---|
| BOM 管理 | 多层物料清单、版本管理、替代料、有效期 |
| 工艺路线 | 工序、工作中心、标准工时 |
| MRP 运算 | 依据销售订单/预测 + BOM + 现有库存 + 在途，运算净需求，输出采购请购与生产建议 |
| 生产订单 | 下达、排程、领料、报工、完工入库 |
| 在制与齐套 | 在制品库存、缺料齐套分析 |

**与其他模块协作**：MRP 生成请购 → 采购；生产领料 → 库存出库；完工入库 → 库存入库；生产成本归集 → 成本核算。

### 3.6 财务管理

| 子模块 | 职责 |
|---|---|
| 总账 GL | 凭证（手工/自动）、记账、期末结转、试算平衡、结账、财务报表（资产负债/利润/现金流） |
| 应收 AR | 应收单、收款单、核销、信用控制、账龄分析 |
| 应付 AP | 应付单、付款单、核销、付款计划、账龄分析 |
| 成本核算 | 库存计价、采购暂估、标准成本/实际成本、生产成本归集与分摊、差异结转 |

**核心机制——业财一体**：业务单据（收货、发货、出入库、核销）按预置"记账规则/事件-科目映射表"自动生成会计凭证，记账凭证保留对源业务单据的反查链路，保障账实一致与可追溯。

### 3.7 系统管理

用户/组织/角色/权限、菜单与按钮权限、数据权限（按组织/部门）、审批流配置、单据编码规则、参数配置、汇率/税率维护、操作日志与审计。

## 4. 关键业务流程

### 4.1 O2C（订单到收款）端到端流程

```
销售订单 ──信用/ATP校验──> 审批 ──> 发货通知 ──> 拣货出库
   │                                              │
   │                                    [库存↓ + 结转销售成本(成本)]
   ▼                                              ▼
开票申请 ──> 销项发票(税务接口) ──> 生成应收AR ──> 收款(银企接口)
                                          │              │
                                    [GL:确认收入]   [AR核销 + GL:银行存款]
```

| 步骤 | 触发的跨模块动作 | 状态流转 |
|---|---|---|
| 销售订单确认 | 库存占用（ATP）、信用占用（AR） | 订单：已审批 |
| 发货出库 | 库存减少、结转销售成本（成本/GL） | 订单：已发货 |
| 开票 | 生成销项发票 → 税务系统；确认收入与销项税（GL） | 订单：已开票 |
| 生成应收 | AR 应收单建立 | 应收：未核销 |
| 收款核销 | 银企收款回单匹配，AR 核销，GL 记银行存款 | 应收：已核销 |

### 4.2 P2P（采购到付款）端到端流程

```
请购(MRP/补货点) ──> 审批 ──> 询比价 ──> 采购订单 ──> 收货
                                          │            │
                                          │    [库存↑ + 采购暂估(成本/GL)]
                                          ▼            ▼
                              发票校验(三单匹配) ──> 生成应付AP ──> 付款(银企接口)
                                          │              │
                                  [冲暂估+进项税(GL)]  [AP核销 + GL:银行存款]
```

| 步骤 | 触发的跨模块动作 | 状态流转 |
|---|---|---|
| 收货 | 库存入库、采购暂估入账（GL 暂估应付） | 订单：已收货 |
| 发票校验 | PO+收货+发票三单匹配；冲销暂估、确认进项税、生成 AP | 发票：已校验 |
| 付款核销 | 银企付款，AP 核销，GL 记银行存款 | 应付：已核销 |

### 4.3 产销存联动（计划-生产-库存-成本）

```
销售订单/预测 ──> MRP运算 ──┬──> 采购建议 ──> P2P ──> 原料入库(库存↑)
   (需求)        (净需求)    │
                            └──> 生产建议 ──> 生产订单
                                              │
                                  领料(库存↓) ──> 报工 ──> 完工入库(库存↑)
                                              │                    │
                                     [生产成本归集(成本)]    [产成品计价入账]
                                              │
                              销售发货(库存↓ + 结转成本) ──> O2C
```

**说明**：MRP 以销售需求为输入，结合 BOM 展开、现有库存与在途，输出采购与生产计划，实现"以销定产、以产定购"；库存是产供销的实物枢纽，所有实物增减集中于库存账；成本模块全程归集采购、生产、销售成本，期末核算产品成本与销售毛利。

### 4.4 跨模块数据流转与状态触发机制

- 采用"业务事件驱动 + 凭证自动生成"：每个关键业务动作发布领域事件（如 `GoodsIssued`、`InvoiceVerified`），下游模块（库存、财务）订阅并执行对应处理。
- 单据状态机统一管理：草稿→已提交→已审批→执行中→已完成→已关闭/已作废，状态跃迁触发后续动作并校验前置条件。
- 财务凭证通过"事件—科目映射规则表"生成，规则可配置，保证业务与财务的自动同步与一致。

## 5. 数据设计

### 5.1 核心实体清单

| 领域 | 核心实体 |
|---|---|
| 主数据 | 物料 Material、客户 Customer、供应商 Supplier、组织 Organization、计量单位 UOM、币种 Currency、会计科目 GLAccount |
| 采购 | 请购单 PurchaseRequisition、采购订单 PurchaseOrder、收货单 GoodsReceipt、采购发票 PurchaseInvoice |
| 销售 | 销售订单 SalesOrder、发货单 Delivery、销售发票 SalesInvoice |
| 库存 | 库存余额 StockBalance、库存事务 StockTransaction、批次 Batch、调拨单 TransferOrder、盘点单 StockCount |
| 生产 | BOM、工艺路线 Routing、生产订单 ProductionOrder、领料单 MaterialIssue、MRP 计划 MRPPlan |
| 财务 | 会计凭证 Voucher、应收单 ARDocument、应付单 APDocument、收付款单 Payment、成本核算单 CostSheet |

### 5.2 ER 概要图

```
   Organization ─┐
                 ├─< (各业务单据均隶属组织)
 Customer ──< SalesOrder ──< Delivery ──> StockTransaction >── Material
                 │                              │
                 └──< SalesInvoice ──> ARDocument ──> Payment
                                            │
 Supplier ──< PurchaseOrder ──< GoodsReceipt ──> StockTransaction
                 │                              │
                 └──< PurchaseInvoice ──> APDocument ──> Payment
                                            │
 Material ──< BOM >── Material(子件)         │
 Material ──< ProductionOrder ──< MaterialIssue ──> StockTransaction
                                            │
 StockTransaction ──聚合──> StockBalance     │
 ARDocument / APDocument / StockTransaction ──触发──> Voucher ──> GLAccount
```

- `StockTransaction`（库存事务）是所有实物增减的统一事务记录，向上汇总为 `StockBalance`。
- `Voucher`（会计凭证）由库存事务、应收/应付、收付款等业务事件统一驱动生成，并反查源单。

### 5.3 主数据设计要点

| 主数据 | 设计要点 |
|---|---|
| 物料 | 全局唯一编码；属性分采购/销售/库存/生产/财务视图，按组织维护差异化属性 |
| 客户/供应商 | 集团统一编码，组织级扩展（信用、付款条件、银行账户）；准入与冻结状态 |
| 组织 | 树形多层级，定义会计主体（法人）边界，作为数据隔离与共享的基准维度 |
| 币种与汇率 | 维护本位币、交易币、汇率类型（即期/月初/期末），支撑多币种折算 |
| 会计科目 | 多维核算（科目+核算项目：客户/供应商/部门/项目） |

### 5.4 数据存储策略

- 交易型核心数据存 PostgreSQL，强一致事务；高频热点主数据 Redis 缓存。
- 大表（库存事务、凭证、单据明细）按组织 + 年月分库分表（ShardingSphere）。
- 历史数据归档：联机库保留近 2 年，历史迁至归档库，报表/检索走 ES + OLAP。
- 单据与凭证不可物理删除，仅逻辑作废 + 留痕，满足审计要求。

## 6. 接口设计

### 6.1 接口设计原则

- 内部模块间优先采用领域事件（异步、最终一致）解耦；强一致场景采用同步 REST/RPC + 分布式事务。
- 对外接口统一经网关，采用 RESTful + JSON，认证用 OAuth2，重要接口加签名验签与幂等控制。

### 6.2 内部模块接口

| 接口 | 提供方 → 调用方 | 方式 | 说明 |
|---|---|---|---|
| 库存可用量查询（ATP） | 库存 → 销售 | 同步 REST | 销售下单时校验可发数量 |
| 出库执行 | 销售/生产 → 库存 | 事件 `IssueRequested` | 发货/领料触发库存减少 |
| 入库执行 | 采购/生产 → 库存 | 事件 `ReceiptConfirmed` | 收货/完工触发库存增加 |
| 库存事务 → 凭证 | 库存 → 财务 | 事件 `StockMoved` | 按映射规则生成成本凭证 |
| 应收生成 | 销售 → AR | 事件 `InvoiceIssued` | 开票后生成应收 |
| 应付生成 | 采购 → AP | 事件 `InvoiceVerified` | 校验通过生成应付 |
| MRP 需求 | 销售 → 生产 | 同步/批量 | 订单与预测作为 MRP 输入 |
| 主数据广播 | 主数据 → 全模块 | 事件 `MasterDataChanged` | 主数据变更刷新缓存 |

### 6.3 外部系统接口

| 外部系统 | 协议 | 数据契约/内容 | 方向 |
|---|---|---|---|
| 银行（银企互联） | 银行直连/银联渠道，HTTPS+证书加签 | 付款指令、余额查询、交易回单、对账文件 | 双向 |
| 税务（金税/电子发票） | 国家电子发票服务平台 API，HTTPS+CA 签名 | 销项开票请求、发票号回写、进项发票查验、税控报送 | 双向 |
| 电商平台（天猫/京东/自建商城） | OpenAPI REST + Webhook | 订单下推、库存上推、发货回传、退款单 | 双向 |
| MES（制造执行） | REST / 消息队列 / OPC-UA | 生产订单下发、工单进度/报工、领料完工回传、设备产量 | 双向 |
| 物流（WMS/TMS/快递） | REST API | 发货单、运单号、签收回传 | 双向 |
| 单点登录/统一身份 | OAuth2 / SAML / LDAP | 用户认证、组织同步 | 接入 |

**契约示例（采购付款指令，ERP → 银行）**：

```json
{
  "requestId": "PAY20260630000123",      // 幂等键
  "orgCode": "L001",                       // 付款法人组织
  "payerAccount": "62220000111122223333",
  "payeeName": "供应商A",
  "payeeAccount": "62220000999988887777",
  "currency": "CNY",
  "amount": 158600.00,
  "purpose": "采购货款-PO20260601",
  "sign": "MD5/RSA签名串"
}
```

返回：受理状态 + 银行流水号；后续通过回单接口异步回写实际支付结果，触发 AP 核销。

## 7. 非功能性设计

### 7.1 性能指标

| 指标 | 目标 |
|---|---|
| 联机交易响应 | 常规单据保存/查询 P95 ≤ 1.5s，简单查询 ≤ 500ms |
| 并发能力 | 支持 ≥ 2000 并发在线用户，核心接口 ≥ 1000 TPS |
| MRP 批运算 | 10 万物料 + 10 万订单行，运算 ≤ 30 分钟 |
| 月结/对账批处理 | 单法人年度结账 ≤ 2 小时 |
| 数据规模 | 单据明细/库存事务/凭证支持十亿级，通过分库分表 + 归档保障 |

优化手段：读写分离、热点缓存、异步化（MQ 削峰）、批处理分片并行、SQL 与索引优化、ES 承接复杂查询。

### 7.2 安全与权限设计（RBAC）

**RBAC 角色模型**：

```
用户 User ──(n:m)── 角色 Role ──(n:m)── 权限 Permission
   │                  │
   └─归属─ 组织/部门    └─ 功能权限(菜单/按钮) + 数据权限(组织/部门/客户范围)
```

| 维度 | 设计 |
|---|---|
| 认证 | 统一登录 SSO，OAuth2 + JWT，支持多因子（MFA） |
| 功能权限 | 角色绑定菜单、按钮、API 操作权限 |
| 数据权限 | 按组织/部门/业务范围行级隔离，多组织数据天然隔离 |
| 字段级 | 敏感字段（成本价、银行账户、信用额度）按角色脱敏/屏蔽 |
| 传输与存储 | 全链路 HTTPS/TLS；敏感数据加密存储；密钥集中管理 |
| 防护 | 防 SQL 注入/XSS/CSRF；接口防重放、限流、验签；越权校验 |

### 7.3 可用性与容灾

- 应用无状态多副本，K8s 自愈与滚动发布，单节点故障不中断服务。
- 数据库主从 + 同城灾备 + 异地灾备（两地三中心），核心库 RPO ≤ 5min、RTO ≤ 30min。
- 系统整体可用性目标 99.9%；关键支付/开票接口具备熔断、降级、重试与人工补偿。
- 全量 + 增量备份策略，定期容灾演练。

### 7.4 可扩展性

- 微服务按领域拆分，可独立部署与扩容；无状态应用支持水平弹性伸缩。
- 业务规则、单据编码、审批流、科目映射均配置化，减少二开。
- 预留扩展字段与插件式扩展点（SPI），新增模块/国别税制按需挂载。
- 开放 API 与事件总线，便于对接新外部系统。

### 7.5 ERP 特有非功能设计

| 特性 | 设计 |
|---|---|
| 多组织 | 集团-法人-业务单元多层组织模型，数据按组织隔离与受控共享，支持组织间交易（内部往来）、合并报表 |
| 多币种 | 交易币/本位币/集团报告币三层，维护多种汇率类型，自动汇兑损益计算与期末重估 |
| 审计追溯 | 单据-库存-凭证全链路反查；关键数据变更留痕（操作人/时间/前后值）；单据不可删除仅作废；日志不可篡改并满足审计留存年限 |
| 期间管理 | 会计期间开关控制，已结账期间禁止业务回写，跨期调整走专用流程 |
