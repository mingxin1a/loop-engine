# 拆分实例参考(示例,不是模板)

> **读这个文件的正确方式**:看"接缝长什么样、裁判怎么配、退回怎么走",建立直觉。
> **不要照抄阶段划分。** 每个实例的阶段只适用于它自己。碰到新任务,回到 SKILL.md 的
> "怎么找拆分线",从你这个任务的实际接缝现推。下面的领域如果和你的任务不同构,它的阶段对你就是错的。

每个实例统一看四列:**环节 → 专责 agent 产出 → 独立裁判 → 不达标退回**。
通用规律(跨所有实例成立,这才是真正要记的):

- 沿"产物形态 / 所需专长 / 依赖断点 / 可独立验证"切。
- 能上**客观闸门**(可执行校验)的环节就别用 AI 估;AI 评审只补客观信号管不到的质量维度。
- **最小退回**:终判不达标只退回出问题的那一段。
- 互不依赖的段并行;有依赖的串行。多 agent 同时改文件时,各用各的 git worktree 隔离。

## 实例 A:软件开发(细粒度版)

总目标示例:做出满足需求、测试通过、可发布的功能。**注意这是细拆的样子,不是必须照搬的层数**——
真实粒度要 match 任务大小:改个小 bug 别拆这么细,做个新子系统可能还要更细(每条往下递归)。

不是一条直线,而是一张 DAG(`──►` 依赖,`∥` 可并行):

```
[需求澄清] ──► [需求规格] ──► [概要架构] ──► [详细设计/接口契约] ──► [任务分解]
                                                                      │
                        ┌──────────────── 按模块并行 ────────────────┤
                        ▼                  ▼                  ▼       │
                  [模块A实现]∥        [模块B实现]∥        [模块C实现]   │
                        │ 各带:编译+类型+lint(客观)+单测+代码评审      │
                        └──────────────► [集成] ◄────────────────────┘
                                           │ 集成测试(客观)
                                           ▼
                              [E2E/验收测试] ──► [安全/性能评审] ──► [文档] ──► [发布前终判]
```

逐节点(产出 / 上游契约 / 独立裁判 / 退回):

| 节点 | 专责 agent 产出 | 吃上游什么(契约) | 独立裁判 | 不达标退回 |
|---|---|---|---|---|
| 需求澄清 | 厘清意图、列开放问题 | 用户原始目标 | 是否消歧、无矛盾(可由用户裁) | 再问 |
| 需求规格 | 可验收的需求条目 | 澄清结论 | 每条是否**可独立验收**(否则没法当裁判) | 重写规格 |
| 概要架构 | 分层 / 组件 / 数据流 | 需求规格 | 设计评审:满足非功能约束?可扩展? | 重设计 |
| 详细设计 | **接口契约/类型/schema** | 概要架构 | 接口是否完备、自洽、可被并行实现 | 补接口 |
| 任务分解 | 模块清单 + 依赖图 | 详细设计 | 是否覆盖全部需求、依赖无环 | 重分解 |
| 模块实现 ×N | 各模块代码 | **只看该模块的接口契约**(隔离上下文) | 编译+类型+lint(客观)+单测+代码评审 | 改本模块 |
| 集成 | 接起各模块 | 各模块产物 + 契约 | 集成测试(客观) | 退回出错模块 |
| 验收/E2E | 端到端用例 | 集成产物 + 需求规格 | E2E 通过率 / 覆盖率(客观) | 退回实现或设计 |
| 安全/性能评审 | 风险清单 | 全量代码 | **多视角裁判**(安全 / 性能 / 可复现各一) | 退回对应环节 |
| 文档 | 使用/接口文档 | 全量产物 | 文档与实现是否一致 | 补文档 |
| 发布前终判 | — | 全部产出 | 需求全覆盖?三者自洽?门禁全绿? | 最小退回到问题节点 |

这一条实例同时示范了多个旋钮怎么拨:
- **拓扑**:前半串行(设计有强依赖),模块实现段并行,集成处汇合 → 典型 DAG。
- **接口契约**(旋钮 4):"详细设计"专门产出接口/类型,正是它让"模块实现"能并行、能各自单独质检。
- **上下文记忆**(旋钮 11):每个模块实现 agent **只看自己模块的契约**,不灌全量代码——隔离防串味、省上下文。并行改文件各自 git worktree。
- **校验强度 + 视角多样性**(旋钮 8/9):能上客观闸门处全用 build/test/typecheck;安全/性能评审用多视角裁判而非同质投票。
- **递归**(旋钮 3):任一"模块实现"若自己就是个大工程,就地对它再跑一遍本流程。
- **最小退回**(旋钮 7):终判挂了只退回出问题的那个节点,不从需求重来。

## 实例 B:写一份文档

总目标示例:产出一份结构完整、图文表一致的报告。

| 环节 | 专责 agent 产出 | 独立裁判 | 不达标退回 |
|---|---|---|---|
| 拆章节(planner) | 大纲 + 每节目标 | 大纲评审(是否覆盖主题、逻辑顺) | 重排大纲 |
| 写文字 | 各节正文 | 文字裁判(清晰、准确、口径一致) | 重写该节 |
| 生成表格 | 数据表 | 数值对账 / schema 校验(客观) | 修表 |
| 生成图表 | 图表(依赖表格数据) | 图表 vs 表格是否对得上 | 重画 |
| 合成(assembler) | 拼成全文,统一风格 | 终判:风格统一?图表与正文一致?有无断层 | 退回对应段 |

要点:图表依赖表格 → 串行;独立段落可并行。表格能用校验脚本当客观裁判。

## 实例 C:研究报告

总目标示例:就某问题产出有据可查、结论可信的报告。

| 环节 | 专责 agent 产出 | 独立裁判 | 不达标退回 |
|---|---|---|---|
| 检索 | 多角度搜来源 | 覆盖度裁判(角度是否全、来源是否够) | 补搜 |
| 阅读/抽取 | 从来源抽事实+出处 | 出处可达性校验(客观)+ 抽取准确性 | 重读 |
| 撰写 | 综合成报告 | 结构与论证裁判 | 重写 |
| 核查 | — | **对抗式核查**:逐条结论找反证,多裁判投票 | 退回撰写/检索 |

要点:核查环节适合多 AI 对抗投票(默认倾向"挑错"),降低单裁判噪声;出处可达性是客观闸门。

## 实例 D:数据管线

总目标示例:从原始数据产出可信的分析结论 / 看板。

| 环节 | 专责 agent 产出 | 独立裁判 | 不达标退回 |
|---|---|---|---|
| 抽取 | 拉取原始数据 | 行数/字段/schema 校验(客观) | 重拉 |
| 清洗 | 去重、补缺、规整 | 数据质量断言(空值率、范围、对账) | 重洗 |
| 建模/分析 | 指标、模型、结论 | 口径核对 + 结果合理性评审 | 重算 |
| 可视化 | 图表/看板(依赖上游) | 图表 vs 数据一致性 + 可读性 | 重画 |

要点:几乎每段都有客观断言可当裁判(schema、对账、质量阈值),优先用。串行依赖强。

## 怎么用这些实例

1. 先判断你的任务和哪个实例**同构**(接缝相似),没有就别硬靠。
2. 借走的是**方法**(沿接缝切、配独立裁判、最小退回、优先客观闸门),不是阶段名。
3. 阶段、裁判、退回路径,都按你这个任务重新写一遍——哪怕和某实例很像,也要现推一次确认。
