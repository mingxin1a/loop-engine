# LoopEngine — 目标驱动的提示词迭代闭环

## 这是什么

用户**不写提示词,只给一个可衡量的目标**。你负责自动跑这个闭环,直到输出达标:

```
目标 ──► 生成/改提示词 ──► 跑出输出 ──► 独立裁判按目标打分 ──► 达标? ──┐
            ▲                                                   │ 否    │ 是
            └──────────── 把裁判反馈喂回去 ◄──────────────────────┘       ▼
                                                                       交付
                          (命中阈值 或 用完预算 才停)
```

它和"我直接帮你做"的本质区别,也是这个 skill 唯一不可妥协的原则:

> **裁判必须独立于生成端。** 出题、答题、判卷如果都是同一个你,你会倾向于自评过高然后提前收手。
> 所以负责"评判/打分"的角色,要和负责"写提示词、跑输出"的角色**分开**——用独立的子 agent、
> 用户本人、或一组测试用例来卡阈值。这是整个循环不会"自己骗自己"的前提。

## 最短路径(90% 的任务读到这里就够)

别被下面 12 个维度吓退。**多数任务只需要这条直路**:

1. **把目标变可判定**:一句话敲定"怎么算达标 + 阈值 + 轮数/预算上限"(详见"启动前"一节)。只问缺的。
2. **默认走 A(轻量内联)**:你在对话里 propose→run→**独立子 agent 判分**→不达标带反馈再来,跑到达标或用完预算。每轮简报一行。
3. **达标就交付四样**:最终提示词 + 输出 + 裁判评分 + 迭代轨迹表。

**只有当任务真的复杂**(能拆成多个互不依赖块、要多 agent 改文件、要多裁判对抗、规模大会爆上下文、不可逆高风险)
**且成本已授权**时,才往下读"维度"和"B. 多智能体编排"。一个贯穿全篇的铁律先记住:**下面所有领域实例都是
示例不是模板,碰到新任务永远从它自身的接缝现推拆法,别套模具**——这条后面不再重复,默认一直生效。

## 引擎的维度(正交旋钮,按任务组合,不是固定流程)

LoopEngine 不是"一条流水线",而是**一组正交旋钮**。每个任务现场拼一套结构出来——拨哪些、拨多深,
由任务本身决定。下面 12 个旋钮分 5 组。**别被数量吓到:绝大多数任务只需拨动其中几个**,其余留默认。

> **维度卫生(先读这条,否则维度会害你)**:多维度只在"正交 + 可操作"时才让结果更优。
> **只拨那些会真正改变你做法的旋钮**;一个维度在当前任务上不改变任何行动,就留默认、连提都别提。
> 把精力花在"凑齐所有维度"而不是"解决任务"上,是最常见的过度工程——精细化是手段,不是目的。

### 组 1 · 结构(怎么切、怎么连)

1. **分解** —— 沿任务的天然接缝把目标切成阶段/角色(见下"怎么找拆分线")。简单任务可不切。
2. **拓扑** —— 切完的段怎么连:串行(有依赖)/ 并行(互不依赖)/ DAG(混合,多数真实任务是 DAG)。
   拓扑独立于"切成什么",是单独的执行形状决策。
3. **递归** —— 一个节点**满足下面信号才**就地展开成它自己的子流水线(分形,深度自适应);否则直接做完。
   判据是**节点的可观察特征,不是"感觉够不够复杂"**——满足任一条才展开:
   ① 这一节点要产出**多种形态产物**(如既要散文又要表格又要代码);
   ② 它内部**有自己的依赖链 / 多个能并行的子块**;
   ③ 它**无法作为单个单元被独立质检**(裁判没法一次性判它达没达标);
   ④ 估算步骤/规模大到塞进单个 agent 会爆上下文或丢轨迹。
   都不满足就**直接做完,别硬展开**(展开本身有协调成本)。**这是"精细"的主引擎**:要更细,
   优先往下递归,而不是往上堆更多顶层维度——但递归与否由上面信号决定,不由体量形容词决定。
4. **接口契约** —— 段与段之间交接的**类型化约定**(上游产出什么字段/格式,下游据此消费)。
   拆得越细,契约越是命门——没有它,细拆只会让交接含糊、彼此返工。

### 组 2 · 收敛(怎么逼近达标)

5. **迭代** —— 对任一目标反复 propose→run→judge。可套在整体,也可只套在某一段上。
6. **收敛判据** —— 何时停:达阈值 / 连续 N 轮无实质改进 / 挖到枯竭(再迭代不再产出新东西)/ 用完预算。
   比"达标或超预算"二选一精细,能避免在已收敛处空转、也避免过早收手。
   **轮数上限可固定也可动态**:最简单是写死 N 轮(达标提前退);要更省,可**先跑一轮拿到首个真实分数,
   按它距阈值的差距估出动态上限**(差得远多给几轮、已接近就少给),再叠加"连续 N 轮提升不足即判枯竭"提前停。
   无论固定还是动态,**都必须有一个绝对硬上限兜底**(见旋钮 10),防止评估失误或反复失败时无限刷。
7. **反馈路由** —— 不达标时退回**哪一环、退多远**。守则是**最小退回**:只退到出问题的段,不推翻全部。

### 组 3 · 质检(凭什么信结果)

8. **校验强度** —— 裁判硬度梯度:`自检 < 单 AI 评审 < 多 AI 对抗投票 < 客观闸门(测试/校验脚本)`。
   越关键、越易自欺的环节用越硬的;能上客观闸门就别用 AI 估。
9. **视角多样性** —— 同一节点用**几个不同视角**的裁判(正确性 / 安全 / 性能 / 可复现 …),而非多个同质裁判。
   和"强度"正交:强度管闸门硬不硬,多样性管能不能照出不同的失败面。

### 组 4 · 资源与状态(用什么喂、状态怎么流)

10. **预算分配** —— 把总预算/算力**自适应倾斜**到正在卡的节点,而不是平均撒。无上限会失控,所以总额永远要封顶。
11. **上下文记忆** —— 每个 agent **看到什么**:共享黑板 vs 隔离上下文、产物如何在段间传递、裁判只看目标+输出。
    细拆后这条决定成败——给多了串味+爆上下文,给少了断链。**默认最小必要上下文**。
    **跨会话/长 loop** 时,把目标、迭代轨迹、当前状态写进持久文件(如 `STATE.md` / run-log),
    别只活在对话里——这样中断后能续跑、也能审计。

### 组 5 · 治理(放多少权给自动化)

12. **自治度 & 人工闸门** —— 这一维管"**哪些决策要人批**",和"做什么/判多严"正交。按风险分级推进:
    `L1 只报告(给方案不动手) → L2 协助(自动改但需人确认) → L3 无人值守(达标即落地)`。
    **遇到高风险或歧义(删数据、改线上、需求本身不清)→ 升到人工闸门,别让 loop 自己拍板**。
    自动化会**放大**好与坏的决策,所以越不可逆的动作,自治度要越低。默认从 L1/L2 起步,信任建立后再放权。

一个任务的最终结构 = 这些旋钮的一组取值(多数留默认)。下面各节讲常用旋钮怎么拨;
要更精细,**首选往下递归(旋钮 3)、收紧契约(旋钮 4)**,而不是发明新顶层维度。

## 启动前:把目标变成"可判定"

循环能不能跑,取决于"达标"能不能被客观判定。开跑前用一两句话和用户敲定下面三样。
**只问缺失的,别把已经清楚的再问一遍。** 如果用户的目标本身已经够明确(比如"全过这组测试用例"),
就直接默认开跑,不要为了仪式感反复确认。

1. **怎么算达标(评判标准)** —— 从下列里识别用户属于哪种,据此选裁判:
   - **有标准答案 / 参考输出** → 裁判按相似度或要点覆盖打分
   - **有打分规则(如 0–100、一组 rubric 维度)** → 裁判按 rubric 逐项给分
   - **有测试用例 / 可执行校验** → 直接跑,通过率就是分数(最客观,优先用)
   - **只能主观判断(文风、创意、观感)** → 由**用户本人当裁判**,每轮把候选交给他选/评
2. **停止条件** —— "达标即停"(给出阈值,如 ≥90 分 / 全过),还是"在 N 轮里尽量做到最好"
3. **预算上限** —— 最多几轮迭代(默认 5 轮)、或最多花多少 token。**永远要有上限**,避免无限刷。

把这三样确认后,简短复述一遍"目标 / 裁判方式 / 阈值 / 上限",然后开跑。

## 怎么跑这个循环

按**任务复杂度**选执行方式,不按用户语气。**默认用 A(轻量内联)**;只有当任务的复杂度
**真的需要**并行/隔离/多裁判,**且**成本被授权时,才升级到 B。

> **判据是复杂度 + 成本,不是"认真不认真"。** "认真做 / 要稳"这类**态度词不参与 A/B 决策**——
> 简单任务哪怕用户说"认真点",也走 A,只在 A 内部把校验强度调高(如单裁判换多视角裁判)。
> 简单/单块/低风险的任务硬上多智能体编排,纯属烧 token + 拖慢。
>
> **该升 B 的复杂度信号(满足几条才考虑,不是沾一条就升):**
> - 交付物能拆成**多个互不依赖的块**,并行能真省时间(单块任务并行无意义)
> - 需要**多 agent 同时改文件**、要 git worktree 隔离
> - 需要**多裁判对抗投票**降噪(单点主观判断风险高、易自欺)
> - 轮数多 / 规模大,内联会爆上下文或丢迭代轨迹
> - 任务**不可逆 / 高风险**,值得用更硬的物理隔离
>
> **成本闸门(独立于复杂度,两者都满足才走 B):** Workflow 很烧 token,按 harness 规则
> **需要用户显式 opt-in**(用户明确要多智能体/workflow、或 ultracode)。即使判定"复杂",
> 也不要静默起 workflow——没授权就先说明代价再问,或先用 A 起步。

### A. 轻量内联(默认)

你在对话里手动驱动,但**裁判仍然独立**——评判这一步交给一个**独立子 agent**(Agent 工具,
不带本轮怎么写提示词的偏向上下文),让它"只拿着目标和输出来打分"。

每一轮做四件事,并向用户**简报本轮**(别闷头刷):

1. **propose** —— 写/改这一版提示词。第一版从目标直接推导;之后每版都要明确"针对上轮裁判的哪条反馈做了什么改动"。
2. **run** —— 用这版提示词跑出实际输出。
3. **judge(独立)** —— 起一个独立子 agent,只给它【目标 + 评判标准 + 本轮输出】,
   让它返回 `{分数, 是否达标, 每条不达标的具体原因和改进建议}`。不要把"我希望它过"这种倾向喂给裁判。
   若标准是测试用例,直接跑用例,通过率即分数。若标准是主观,把候选交给用户裁决。
4. **decide** —— 达标或用完预算 → 停并交付;否则把裁判的具体反馈带进下一轮的 propose。

用一个小表持续展示进度,让用户随时能叫停:

```
轮次  提示词改动要点          分数/阈值     裁判主要意见
 1    初版,直接由目标推导      62 / 90      漏了边界情况;语气太啰嗦
 2    增加边界约束+精简语气     88 / 90      还差一个格式要求
 3    补格式硬约束             93 / 90  ✅  达标
```

#### 一个完整微样例(默认 A 模式长什么样)

**用户目标**:"给我一句电商大促的英文 push 文案,要短、有紧迫感、含明确行动号召。"
**敲定可判定**(一句话):rubric 0–100,四项各 25——①≤12 词 ②含紧迫感 ③含明确 CTA ④无语病/自然;**阈值 ≥85,上限 3 轮**。

> **R1 · propose**(我写):"Write a short English push notification for an e-commerce sale, with urgency and a call to action."
> **run** → `Big sale is happening now, don't miss out on great deals, shop today!`(15 词)
> **judge(独立子 agent,只给它 rubric+这句输出)** → `{score:70, pass:false, feedback:["超 12 词(15)","CTA 偏弱,'shop today' 不够具体","urgency 靠 'now' 单点,可更强"]}`
>
> **R2 · propose**(针对 R1 三条反馈改:压词数 + 强 CTA + 强紧迫):"…限定 ≤10 词,CTA 用祈使句带具体动作,紧迫感用倒计时词…"
> **run** → `Flash sale ends at midnight—grab 50% off now!`(8 词)
> **judge** → `{score:88, pass:true, feedback:[]}` ✅ 达标,停。

**交付四样**:最终提示词(R2 那版)+ 输出那句 + 裁判 88/85 + 上面这张轨迹。
注意三个细节:裁判**全程只看到 rubric+输出**(没看到"我希望它过");每轮 propose 都**点名改了 R1 的哪条**;
**达标即停**没有硬刷满 3 轮。这就是 90% 任务的真实形态——无需 Workflow、无需多 agent。

### B. 多智能体编排(复杂度需要 + 成本已授权时)

当任务复杂度确实需要(上面那些信号)、且成本已授权时,用 **Workflow** 编排,
把生成与评判彻底物理隔离,并可并行多裁判投票降低单裁判噪声:

- **propose 阶段**:一个 agent 产出/改写候选提示词
- **judge 阶段**:多个**互不通气**的 agent 各自独立按 rubric 打分(对抗式,默认倾向"挑毛病"),
  取多数/均值;主观任务则交用户
- 不达标 → 汇总裁判反馈喂回 propose → 下一轮
- 命中阈值 或 用完 `budget` → 停

Workflow 的 `budget`、`pipeline`、`parallel` 天然适配"propose→judge 多轮、判卷并行投票"。
**复杂度够 + 用户已授权**才走这条(它会消耗较多 token);任一不满足就用 A。

## 分工编排:多 agent 各管一块(可与上面的 loop 叠加)

**这条机制和领域无关**——只要交付物能拆成几段、每段需要不同专长,就适用。不要让一个 agent
从头包到尾:拆成**流水线,每段一个专责 agent**,每段各自带局部质检,最后合成并对总目标做一次终判。

> ⚠️ 拆法用下面"怎么找拆分线"从**当前任务的接缝**现推(`references/instances.md` 的领域实例只是参照,
> 别套阶段)——这就是最短路径里那条铁律的落地:套错模具会逼任务变形迁就阶段,比不拆更糟。

### 怎么找拆分线(沿任务的天然接缝切,不靠记例子)

按下面几条"哪里变了就在哪切",几条会指向同一处,那就是结实的接缝:

- **产物形态变了** —— 输出物种类换了就是一条缝(散文 ↔ 表格 ↔ 图表 ↔ 代码 ↔ 数据集)。
- **所需专长/角色变了** —— 需要的能力换了就是一条缝(发散设计 ↔ 严谨实现 ↔ 挑错验证 ↔ 润色审校)。
- **出现依赖断点** —— 必须先有 A 才能做 B,A 与 B 之间就该断开(B 吃 A 的产物)。
- **能被独立验证** —— 每段必须能**单独判定达没达标**;如果一段没法单独质检,说明没切对,要么并进相邻段、要么再拆。

**收敛到什么粒度**:切到"每段 = 单一目标 + 明确输入/输出接口 + 可独立质检"为止。
太碎 → 协调和合成成本爆炸、还容易丢上下文;太粗 → 没法专责、也没法局部质检。**粒度本身要match任务**,
不是越多 agent 越好。简单任务就别硬拆——一段到底配一个 loop 往往就够。

### 递归:段内若仍复杂,再展开成子流水线

拆出来的某一段如果命中旋钮 3 的展开信号(产出多形态 / 内部有依赖链或可并行子块 / 没法作为单个单元
被独立质检 / 大到会爆上下文),**不要硬塞给一个 agent**,而是对这一段再跑一次本套引擎
(它自己的 planner→专责段→裁判→合成)。都不命中的段就直接做完。这样整个结构是**分形的**:
深度随这些信号自适应,而不是按"感觉复杂"拍、也不被某个固定层数框死。最硬的那条判据就是那句——
**这一段能不能被单独、清晰地质检;不能,就往下展开或重新划线**。

### 形态

```
总目标 ──► planner 拆块 ──► [块1 专责agent]→[块2]→[块3] ──► assembler 合成 ──► 终判 vs 总目标
                                每块各带局部裁判判到达标          不达标 → 只退回出问题的那块重做
```

### 怎么编排

1. **planner 先拆块** —— 一个 agent 把总目标分解成几块,并标出**块之间的依赖**:
   - 后面的块要吃前面块的产物(图表依赖表格数据 / 结论段依赖前文)→ **串行**,前一块产物作为后一块输入
   - 块之间互不依赖(几个独立段落、独立配图)→ **并行**,更快
   - 混合就混合:能并行的并行,有依赖的串行。
2. **每块一个专责 agent** —— 给它清晰的"本块目标 + 本块达标标准 + 上游产物(若有)"。
   每块可以各自跑一个小 loop(propose→run→judge),由**局部裁判**判到本块达标再往下游传。
3. **assembler 合成** —— 一个独立 agent 把各块拼成完整交付物,负责衔接、统一风格/口径、消重。
4. **终判(独立)** —— 对**整篇**按总目标判一次:各块是否齐、风格是否统一、图表和正文数据是否对得上、
   有没有断层。不达标时**只把出问题的那一块退回重做**,不推翻全部。

### 两层裁判,职责不同

- **局部裁判**:只管"这一块本身好不好"(这段文字够不够清楚、这张表数据对不对)。
- **终判**:管"拼起来作为一个整体是否满足总目标",尤其是跨块的一致性(图表是否反映表格、术语是否统一)。

裁判依旧独立:planner / 各执行 agent / 各裁判 / assembler 是分开的角色,谁都不判自己的卷。

### 优先用"客观裁判",AI 评审只补空隙

裁判不一定要"另起一个 AI 来判"。很多领域有 **0 噪声的客观信号,能直接当裁判,就别用 AI 估**:

- 软件开发:`build` 过不过、单元/集成测试绿不绿、`typecheck` / lint 干不干净、覆盖率达没达标
- 数据/文档:schema 校验、数值对账、链接可达性、格式校验脚本
- 任何有"标准答案 / 可执行校验"的环节:直接跑,通过率即分数

**先用客观信号卡死硬指标,AI 评审只补客观信号管不到的质量维度**(架构合理性、可读性、文风、洞察深度)。
这样裁判既客观又省钱,也最难被"自评过高"。

### 需要参考已拆好的例子时

`references/instances.md` 收了几个已经拆好的领域实例(软件开发、写文档、研究报告、数据管线),
**仅供找感觉和对照**。读它是为了理解"接缝长什么样",不是为了照抄阶段——用之前先确认你的任务
是否真的同构。并行/隔离等工程细节(如多 agent 同时改文件用各自的 git worktree)也记在那。

### 工程对应

这套"串行流水线 + 每段独立质检 + 合成 + 终判"用 **Workflow 的 `pipeline`** 直接对应:
每块作为一个 stage 顺着走,块与块之间不必互相等待;能并行的块用 `parallel`;
要并行改文件时给 agent 加 `isolation: 'worktree'`。
因此当**复杂度够、成本已授权**(见"B. 多智能体编排"的判据)时,用 Workflow 编排最顺。
但**这不是"凡多块就上 Workflow"**:块少、依赖简单、低风险时,在对话里内联串起来就够(走 A),裁判仍要独立。

**可运行的最小示例**:`references/example-loop.workflow.js` 是一个真实跑过的硬循环
(propose → 独立 judge 按 schema 打分 → 反馈回填 → 达阈值或用完轮数才停)。它演示
"循环由代码 `for` 强制、不靠模型自觉"。换任务时改 RUBRIC / SCORE_SCHEMA / THRESHOLD /
propose 提示词即可——**它是示例不是模板,别照搬其中的 ERP 措辞**。

## 交付什么

收敛后,交给用户**这四样**,别只丢一个最终提示词:

1. **最终提示词**(可直接复用的那一版)
2. **它跑出的输出**(证明确实达标)
3. **裁判的最终评分**(对照阈值)
4. **迭代轨迹**(上面那张进度表)—— 让用户看到"为什么是这版、改了什么才达标"

如果用完预算仍没达标,**如实说**:给出最好的一版、差在哪几条、再加多少预算大概能补上,
**不要为了"完成"而把没达标说成达标**。

## 容易踩的坑

- **裁判被污染**:把"我觉得这版不错"塞给裁判,它就会附和。裁判只该看到目标+标准+输出。
- **目标不可判定就硬跑**:会原地打转。宁可先花一句话把标准敲成可判定的。
- **没有上限**:一定设轮数/预算上限,达不到也要停下来如实汇报。
- **过拟合到一个例子**:如果标准是测试用例,别让提示词只为通过这几条而写死;留意泛化。
- **套模具拆任务**:直接套"架构→开发→测试""文字→表格→图表"是最常见的错(详见最短路径那条铁律)。
- **拆得过细**:agent 越多协调和合成成本越高、越容易丢上下文。粒度要 match 任务,简单任务一段到底。
- **把改动说不清**:每轮都要能说出"针对哪条反馈改了什么",否则就是在瞎试。
